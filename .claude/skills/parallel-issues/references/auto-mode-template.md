# AUTO Mode Worker Prompt Template

This template is the common prefix injected into every worker prompt generated by `/parallel-issues`.
It ensures each worker is **self-contained** — able to operate after context clear with zero external state.

---

## Template Structure

```
┌─────────────────────────────┐
│  Common Prefix (this file)  │  ← Same for all workers
├─────────────────────────────┤
│  Issue-Specific Suffix      │  ← Generated per-issue in Phase 3
└─────────────────────────────┘
```

---

## Common Prefix

Paste the following block verbatim at the start of every worker prompt.
Replace only `{{ISSUE_NUMBER}}` and `{{ISSUE_TITLE}}` placeholders.

````markdown
AUTO: Issue #{{ISSUE_NUMBER}} 구현 — {{ISSUE_TITLE}}

You are implementing a single GitHub issue in an isolated git worktree.
Follow the AUTO mode cycle below until the escape condition is met.

## AUTO Mode Cycle

```
PLAN → ACT → EVAL → (loop if Critical > 0 OR High > 0) → /ship
```

Maximum 3 iterations. If escape condition is not met after 3 iterations, stop and report.

### Iteration Flow

#### 1. PLAN Phase
1. Read the issue details below carefully
2. If codingbuddy MCP is available:
   - Call `parse_mode` with your task prompt
   - Call `analyze_task` for risk assessment
   - Call `generate_checklist` for target files
3. Identify the minimal set of changes needed
4. List files to create/modify
5. Call `update_context` with your plan decisions

#### 2. ACT Phase
1. If codingbuddy MCP is available, call `parse_mode` with "ACT: <task>"
2. Implement changes following TDD:
   - **Core logic:** Write failing test → implement → verify pass → refactor
   - **Config/YAML:** Direct implementation → verify no regressions
   - **Documentation:** Direct writing → proofread
3. Run existing tests to verify no regressions:
   ```bash
   yarn workspace codingbuddy test 2>&1 | tail -20
   ```
4. Call `update_context` with progress notes

**SubAgent Parallelization (for Complex issues):**

If the issue is Complex (8+ files, logic implementation), dispatch specialist subAgents:

```
Agent(subagent_type="general-purpose", run_in_background=true,
  prompt="[specialist task description]")
```

Recommended specialists by task type:
- **Architecture decisions:** architecture-specialist
- **Test strategy:** test-strategy-specialist
- **Code quality:** code-quality-specialist
- **Security concerns:** security-specialist

#### 3. EVAL Phase
1. If codingbuddy MCP is available, call `parse_mode` with "EVAL: <task>"
2. Run full test suite and capture results
3. Review your changes against acceptance criteria
4. Classify findings by severity (see eval-criteria.md):

| Severity | Description | Action |
|----------|-------------|--------|
| **Critical** | Broken tests, security vulnerability, data loss risk | Must fix before shipping |
| **High** | Missing acceptance criteria, incorrect behavior | Must fix before shipping |
| **Medium** | Code style issues, missing edge case tests | Note in PR description |
| **Low** | Minor improvements, optional enhancements | Note in PR description |

5. Call `update_context` with findings and recommendations

### Escape Condition

```
IF Critical == 0 AND High == 0:
  → Run /ship {{ISSUE_NUMBER}}
  → DONE

IF Critical > 0 OR High > 0:
  → Return to ACT phase
  → Fix identified issues
  → Re-run EVAL

IF iteration >= 3 AND (Critical > 0 OR High > 0):
  → STOP
  → Report remaining issues
  → Do NOT run /ship
```

### /ship Integration

When escape condition is met, run:

```
/ship {{ISSUE_NUMBER}}
```

This will:
1. Create a branch (if not already on one)
2. Stage and commit changes
3. Push to remote
4. Create a PR linked to issue #{{ISSUE_NUMBER}}
5. Run CI checks

### Important Rules

1. **Stay in your worktree.** Do not modify files outside your assigned worktree.
2. **One issue only.** Focus exclusively on issue #{{ISSUE_NUMBER}}.
3. **No manual merges.** Do not merge master or other branches into your worktree.
4. **Test before shipping.** Always run tests before `/ship`.
5. **Report failures.** If you cannot meet acceptance criteria, report what failed and why.
````

## Issue-Specific Suffix Template

Append the following after the common prefix, filled with actual issue data:

````markdown
---

## Your Task

**Issue:** #{{ISSUE_NUMBER}} — {{ISSUE_TITLE}}
**Complexity:** {{COMPLEXITY}}
**Labels:** {{LABELS}}

### Goal
{{PURPOSE — extracted from issue body}}

### Target Files
{{FILE_LIST — extracted from issue body, one per line with dash prefix}}

### Acceptance Criteria
{{CRITERIA — extracted from issue body, checkbox format}}

### Technical Details
{{DETAILS — extracted from issue body or first comment}}

### Constraints
- This issue runs in parallel with other issues in the same wave
- Your worktree is isolated — no merge conflicts possible if you stay within target files
- Do not modify files outside the target file list unless absolutely necessary
- If you need to modify a shared file not in your list, STOP and report
````

## Prompt Length Considerations

Worker prompts can be long (2000+ characters). When sending via tmux:

1. **Preferred:** Write prompt to a temp file, then use file-based input:
   ```bash
   PROMPT_FILE="/tmp/parallel-issues/prompt-${ISSUE}.md"
   cat > "$PROMPT_FILE" <<'PROMPT_EOF'
   [full worker prompt]
   PROMPT_EOF

   # Send as a single command
   tmux send-keys -t "$PANE" "$(cat $PROMPT_FILE)" Enter
   ```

2. **Alternative:** If tmux send-keys truncates, use clipboard:
   ```bash
   cat "$PROMPT_FILE" | pbcopy  # macOS
   tmux send-keys -t "$PANE" "$(pbpaste)" Enter
   ```

3. **Fallback:** Split into multiple send-keys calls with short delays:
   ```bash
   # Not recommended — can cause input ordering issues
   ```
